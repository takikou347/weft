## 1. プロジェクト概要

### 1.1 背景・目的

個人の「予定・記録・お金」は現在、カレンダーアプリ・日記アプリ・家計簿アプリ・タスク管理ツールに分断されている。既存サービスは家計簿アプリはお金だけ、日記アプリは記録だけ、Notionは自由すぎて設計を強制しない、という状態にあり、「計画 → 当日の記録 → 振り返り → お金」が一つの時間軸で繋がるサービスは空白地帯である。

本プロジェクトは、**「人生を経営する」** をコンセプトに、個人の記録を核として、グループ(友人・家族・サークル)や組織(業務)へ選択的に共有・拡張できる一元管理アプリを個人開発する。

### 1.2 コンセプト

- **個人が起点**: すべてのデータは個人のものとして生まれ、デフォルト非公開
- **共有は「差し出す」行為**: 交換日記のように、自分の記録の中から選んだものだけを相手に見せる
- **カレンダーが背骨**: すべての記録は時間軸に紐づき、カレンダーから辿れる
- **業務管理の手法を個人に**: 家計簿=個人の損益、プロジェクト予実、タスク管理を個人の生活に適用
- **アナログ文化の感性**: 交換日記・回覧板・ギフト帳簿といった日本のアナログ文化を意匠・体験の参照点とする

### 1.3 ターゲットユーザー

| 優先度 | ユーザー像 | 主な利用シーン |
|---|---|---|
| 主 | 記録好きの個人 | 日記、家計簿、予定管理 |
| 主 | 小規模グループ(3〜10人) | 旅行・イベントの計画と思い出共有、立替精算 |
| 副 | 小規模チーム・組織 | プロジェクトのタスク・予実・ナレッジ管理 |

### 1.4 差別化ポイント

1. 計画(カレンダー)・記録(日記)・お金(収支)が双方向リンクで一体化した唯一の構成
2. 「共有されたものだけ見える」選択的共有モデル(Slack型の全体公開とは異なる)
3. 個人・グループ・組織を同一アプリ・同一データモデルで扱える
4. チャット非搭載により法的リスクを抑えつつ、回覧板型のオープンな相互作用を提供

## 2. 用語定義

| 用語 | 定義 |
|---|---|
| スペース | データの共有先となる単位。personal / group / organization / project の4タイプ |
| 個人スペース | ユーザー登録時に自動作成される本人専用スペース。全記録の発生源 |
| グループ | 友人・家族・サークル等の共有スペース |
| 組織 | 業務管理用のスペース。配下にプロジェクトを持てる |
| プロジェクト | 組織配下の作業単位スペース。タスク・予実・ナレッジを持つ |
| アイテム | 記録の実体(予定・日記・収支・タスク・ドキュメント・写真等)の総称 |
| 共有 | アイテムへの参照を特定スペースに付与する行為。データの移動ではない |
| 双方向リンク | アイテム同士の相互参照。リンク元・リンク先の両方から辿れる |

## 4. 全体構造

### 4.1 スペースモデル

```
ユーザー
 └ 個人スペース(自動作成・削除不可)
    ├ 自分のアイテム(デフォルト非公開)
    └ 参加スペース一覧
       ├ グループA(友人)
       ├ グループB(家族)
       └ 組織X
          ├ プロジェクトX-1
          └ プロジェクトX-2
```

- スペースは `type` 属性を持つ単一エンティティ(personal / group / organization / project)
- project は organization を親に持つ。それ以外は親を持たない
- 個人スペースはユーザーと1対1で、退会時のみ削除される

### 4.2 可視性モデル(本アプリの核)

| 原則 | 内容 |
|---|---|
| デフォルト非公開 | アイテムは作成者本人にしか見えない |
| 共有=参照付与 | アイテムをスペースに共有すると、そのスペースのメンバーに見える |
| 共有解除で消える | 共有を解除すると相手からは見えなくなるが、元データは作成者に残る |
| オープン型のみ | スペース内で共有されたものは、そのスペースの全メンバーに等しく見える。特定個人だけに見せる1対1共有は提供しない |
| 重ね合わせ表示 | 個人スペースのカレンダーには「自分の予定+参加スペースに共有された予定」がレイヤーとして重なって表示される |

### 4.3 スペースタイプ別の利用可能機能

| 機能 | personal | group | organization | project |
|---|---|---|---|---|
| カレンダー | ○ | ○(共有分) | ○(共有分) | ○(共有分) |
| 日記・記録 | ○ | ○(共有分) | − | ○(ナレッジとして) |
| 収支 | ○(家計簿) | ○(立替精算) | ○(損益) | ○(予実) |
| アルバム | ○ | ○ | − | − |
| コメント・リアクション | − | ○ | ○ | ○ |
| タスク | ○(個人タスク) | − | − | ○ |
| 予実 | − | ○(イベント予算) | ○(集約表示) | ○ |
| ナレッジ | − | − | − | ○ |

## 5. 機能要件

### F-01 認証・アカウント

- F-01-1 メールアドレス+パスワードでの登録・ログイン(Supabase Auth)
- F-01-2 Google等のOAuthログイン(少なくとも1プロバイダ)
- F-01-3 登録完了時に個人スペースを自動作成
- F-01-4 プロフィール(表示名・アイコン)の設定
- F-01-5 退会機能(全データ削除。共有先からも消えることを明示)

### F-02 スペース管理

- F-02-1 グループ・組織の作成(作成者がowner)
- F-02-2 組織内でのプロジェクト作成
- F-02-3 招待リンクによるメンバー招待(有効期限付きトークン)
- F-02-4 ロール管理: owner / admin / member(§7参照)
- F-02-5 メンバーの退出・除名。退出者の共有アイテムの扱いは「共有維持(本人が解除可能)」を既定とする
- F-02-6 スペースの設定(名前・アイコン・カバー・テーマカラー)

### F-03 カレンダー(メイン機能)

- F-03-1 月表示・週表示・リスト表示
- F-03-2 予定の作成・編集・削除(タイトル、日時、終日フラグ、場所、メモ、繰り返し)
- F-03-3 レイヤー表示: 自分の予定+参加スペースごとの共有予定を色分けし、レイヤー単位で表示ON/OFF
- F-03-4 予定からの派生導線: 予定詳細から「この日の日記を書く」「収支を記録する」「タスクを作る」へ1タップで遷移し、自動で双方向リンクを生成
- F-03-5 カレンダー上に日記・収支・写真の存在をドット等で示し、日付タップでその日の全記録を一覧表示(「その日ページ」)

### F-04 日記・記録

- F-04-1 日付に紐づく記録の作成(タイトル任意、本文、写真添付、タグ)
- F-04-2 1日複数件の記録を許容(日記・メモを同一機能でカバー)
- F-04-3 テンプレート機能(旅行記録・イベント記録等の項目セット)
- F-04-4 装飾要素: 用紙・フレーム・スタンプ等の基本セット
- F-04-5 下書き保存

### F-05 収支記録

- F-05-1 収入・支出の記録(金額、日付、カテゴリ、メモ、レシート写真任意)
- F-05-2 カテゴリのカスタマイズ
- F-05-3 月次集計(カテゴリ別・収支合計)と簡易グラフ
- F-05-4 予定・日記・プロジェクトへのリンク(例: 旅行の予定に交通費を紐づけ)
- F-05-5 組織スペースではプロジェクト別に収支をロールアップし損益表示

### F-06 選択的共有

- F-06-1 任意のアイテムを、自分が参加するスペースへ共有(複数スペースへの同時共有可)
- F-06-2 共有時の範囲指定: アイテム単位を基本とする。日記は本文全体、収支は明細単位
- F-06-3 共有の解除(作成者のみ可能)
- F-06-4 共有状態の可視化: アイテムに共有先バッジを表示し、誰に見えているかを常に確認できる
- F-06-5 共有時に通知(共有先スペースのフィードに掲載)

### F-07 グループ機能

- F-07-1 共有カレンダー: メンバーが共有した予定の一覧・重ね合わせ表示
- F-07-2 思い出アルバム: 共有された写真・日記を時系列・イベント別に閲覧
- F-07-3 回覧板(フィード): スペース内の共有・更新を時系列で表示するオープンな掲示面
- F-07-4 コメント: 共有アイテムへのコメント(スペース全員に可視。非公開返信・1対1メッセージは提供しない)
- F-07-5 リアクション: スタンプによるリアクション
- F-07-6 立替精算: イベントに紐づく立替の記録、割り勘計算、誰が誰にいくら払うかの精算案提示(送金機能は持たず、記録と計算のみ)

### F-08 組織・プロジェクト機能

- F-08-1 プロジェクト管理: プロジェクトの作成、期間・概要・ステータス
- F-08-2 タスク管理: タスクの作成、担当者割当、期限、ステータス(未着手/進行中/完了)、カンバンまたはリスト表示
- F-08-3 予実管理: プロジェクト予算の登録、実績(収支記録)の集計、差異表示
- F-08-4 ナレッジ共有: 議事録・作業記録・ノウハウをドキュメントとして共有。共有されたものだけがプロジェクトメンバーに見える
- F-08-5 組織ダッシュボード: 配下プロジェクトの予実・進捗の一覧
- F-08-6 ナレッジとタスク・予定の双方向リンク(文脈付きナレッジベース)

### F-09 双方向リンク

- F-09-1 アイテム編集画面から他アイテムへのリンクを作成(検索して選択)
- F-09-2 リンクされた側にも自動でバックリンクが表示される
- F-09-3 F-03-4 の派生導線で作成されたアイテムには自動リンクを生成
- F-09-4 リンクはアイテム詳細画面に一覧表示され、タップで遷移
- F-09-5 閲覧権限のないリンク先は表示しない(存在自体を見せない)

### F-10 装飾・デザイン

- F-10-1 スペースごとのテーマ(カラー・カバー)
- F-10-2 日記・アルバムの装飾セット(用紙・フレーム・スタンプ)。第1弾は無料の基本セットのみ
- F-10-3 交換日記・回覧板を想起させるUIモチーフ(フィードを「回覧板」として演出する等)

### F-11 通知

- F-11-1 アプリ内通知のみ(共有された、コメントされた、リアクションされた、タスクを割り当てられた、精算が登録された)
- F-11-2 メール送信は認証系(登録確認・パスワードリセット)と招待のみ
- F-11-3 プッシュ通知は第1弾スコープ外(PWA化と合わせて将来検討)

## 6. データモデル案

### 6.1 主要テーブル

| テーブル | 主なカラム | 備考 |
|---|---|---|
| profiles | id(=auth.users.id), display_name, avatar_url | Supabase Auth連携 |
| spaces | id, type, name, parent_space_id, settings(jsonb), created_by | type: personal/group/organization/project。parentはprojectのみ使用 |
| space_members | space_id, user_id, role, joined_at | role: owner/admin/member |
| invitations | id, space_id, token, expires_at, created_by | 招待リンク |
| items | id, type, owner_id, origin_space_id, occurred_on, title, body, payload(jsonb), created_at | type: event/diary/expense/task/document/photo。typeごとの固有属性はpayloadに格納 |
| item_shares | item_id, space_id, shared_by, shared_at | このレコードがある物だけ他人に見える |
| links | item_id_a, item_id_b, created_by | 双方向リンク(無向) |
| comments | id, item_id, space_id, author_id, body | どのスペース文脈のコメントかを保持 |
| reactions | item_id, space_id, user_id, emoji | |
| projects_meta | space_id(=project), status, start_on, end_on, budget_total | プロジェクト固有情報 |
| budgets | id, space_id, category, planned_amount, period | 予実の「予」側 |
| settlements | id, space_id, event_item_id, payer_id, amount, participants(jsonb), status | 立替精算 |
| notifications | id, user_id, type, payload(jsonb), read_at | アプリ内通知 |

### 6.2 設計原則

- アイテムは単一テーブル+type+payload(jsonb)で開始し、複雑化した種別のみ将来分割する
- 「共有」は item_shares への行追加のみ。データ本体は移動・複製しない
- 収支の集計はビューまたはRPCで実装(月次集計、プロジェクト別ロールアップ)
- 写真は Supabase Storage に保存し、items(type=photo) がメタデータを持つ

### 6.3 RLS(行レベルセキュリティ)方針

すべての items 系テーブルに対し、閲覧許可を次の一本のルールで定義する:

> **自分が作成者である** または **item_shares 経由で共有されたスペースのメンバーである** 場合のみ閲覧可

- 更新・削除は作成者のみ(タスクのステータス変更のみ担当者にも許可)
- 共有(item_shares への insert)はアイテム作成者かつ共有先スペースのメンバーであること
- spaces / space_members はメンバーのみ閲覧可
- organization メンバーであっても、配下プロジェクトのメンバーでなければプロジェクト内アイテムは見えない(プロジェクト単位の分離)

## 8. 非機能要件

### 8.1 プラットフォーム・技術

| 項目 | 内容 |
|---|---|
| 提供形態 | レスポンシブWebアプリ(モバイルファースト)。将来PWA化 |
| 言語 | TypeScript(フロント・サーバー共通) |
| フロントエンド | Next.js(App Router)+ Tailwind CSS。ホスティングはVercel |
| バックエンド | Supabase(Auth / Postgres / RLS / Storage) |
| スキーマ管理 | Supabase CLI によるマイグレーション管理(SQLをGit管理し、手動でのDB変更を禁止) |
| テスト | Vitest(ユニット)+ Playwright(E2E)。RLSの分離テストを必須とする |
| CI/CD | GitHub Actions(テスト・型チェック・マイグレーション検証)+ Vercel自動デプロイ |
| 対応ブラウザ | iOS Safari / Android Chrome / PC Chrome・Edge の最新版 |

**技術選定の妥当性(長期開発の観点)**

- 本アプリの核である「選択的共有」はRLSによるアクセス制御そのものであり、RLSを中核機能とするSupabaseと要件が一致している
- データはすべて標準PostgreSQLに格納されるため、将来Supabaseから自前Postgres等へ移行する退路が確保されている(ロックインが浅い)
- Next.js+VercelはPWA化・ISRに対応し、第1弾から将来拡張まで同一スタックで通せる
- AI(Claude Code)主導の開発において、Next.js/Supabase/Tailwindは学習データ・事例が最も豊富な組み合わせであり、生成コードの品質が安定しやすい

### 8.2 無料枠制約への対応

| リスク | 対応 |
|---|---|
| Supabase Storage容量(写真) | アップロード時にクライアント側で圧縮・リサイズ。1ファイル上限設定。容量逼迫時の告知導線を用意 |
| DB容量・帯域 | payloadの肥大化を監視。一覧APIはページネーション必須 |
| Vercel無料枠 | ISR・キャッシュを活用し関数実行を抑制 |
| Supabase無料プランの自動一時停止 | 約1週間アクティビティがないとプロジェクトが一時停止され、90日を超えると復元が困難になる。GitHub Actionsのcronで定期ヘルスチェックを送信して停止を防止し、一般公開・利用増加後はProプランへの移行を検討する |

### 8.3 性能

- 主要画面(カレンダー月表示・フィード)の初期表示は3G相当でも3秒以内を目標
- カレンダーの月切り替えは体感即時(先読みキャッシュ)

### 8.4 セキュリティ・プライバシー

- 全アクセス制御はRLSで実施し、クライアント側の出し分けに依存しない
- 共有状態は常にUIで可視化し、意図しない公開を防ぐ(F-06-4)
- 退会時は本人起点の全アイテム・共有・写真を削除
- バックアップ: Supabaseの自動バックアップに依存(無料枠範囲)。重要データのエクスポート機能を将来検討

## 9. 運営・法務要件

| 項目 | 方針 |
|---|---|
| チャット・DM | 非搭載。クローズドな通信の媒介を行わないことで電気通信事業の届出を不要とする。コメント等はすべてスペース全員に可視のオープン型とし、非公開返信・1対1メッセージ機能は今後も追加しない |
| 電気通信事業法 | 上記方針により媒介非該当(電子掲示板型)として整理。機能追加時は都度この観点でレビューする |
| 個人情報保護法 | プライバシーポリシーを整備。取得情報(メール、プロフィール、投稿内容)と利用目的を明示 |
| UGCライセンス | 利用規約に、投稿コンテンツの権利はユーザーに帰属し、運営はサービス提供に必要な範囲の利用許諾を受ける旨を明記 |
| 収益化 | 当面は無料・広告なしで運営し、法的露出を最小化。収益化検討時に規約・体制を再整備 |
| 精算機能 | 送金・決済は行わず記録と計算のみ(資金移動業に該当しない範囲に限定) |
| 年齢 | 未成年の利用を想定した表現・コンテンツ方針を利用規約に記載 |

## 12. リスクと対応

| リスク | 影響 | 対応 |
|---|---|---|
| スコープ過大で完成しない | 高 | フェーズ制を厳守し、P2完了時点で「個人アプリ」として成立させる |
| 共有モデルの複雑化 | 中 | 可視性ルールを§4.2の5原則に固定し、例外を作らない |
| 写真によるストレージ逼迫 | 中 | 圧縮・上限・監視(§8.2) |
| 機能追加による法的該当性の変化 | 中 | 機能追加時に§9の観点で都度セルフレビュー |
| 利用者が集まらない | 中 | まず自分と身近なグループで価値検証。市場投入は後 |
