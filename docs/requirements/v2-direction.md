> スペースは企業管理のイメージで、個人/企業をアカウント切替のように行き来したい」

## 1. コンセプトの再定義(案)

**「スケジュールを軸に、個人と組織の記録を貯めるプラットフォーム」**

- **コア = カレンダー/スケジュール**(予定・タスク)。ログイン後の起点をカレンダーにする
- **拡張機能(モジュール)**: 家計簿・日記・写真・文書などは、ワークスペースごとに
  有効/無効を選べる「モジュール」として位置づけ直す
- データがワークスペースに貯まっていくこと(蓄積)を価値の中心に置く

### モジュール候補

| モジュール | 現行機能との対応 | 対象 |
|---|---|---|
| カレンダー/タスク | 実装済み(コア。無効化不可) | 全ワークスペース |
| 家計簿 | 実装済み(家計) | 個人 |
| 日記・写真 | 実装済み | 個人・グループ |
| 文書 | 実装済み(プロジェクト文書) | 全ワークスペース |
| **キャリア(履歴書・職務経歴書)** | **新規**: 学歴・職歴・資格・スキルを構造化して蓄積し、履歴書/職務経歴書として出力(PDF等) | 個人 |
| 精算 | 実装済み | グループ |
| 予実 | 実装済み | プロジェクト |
| 資産・投資・勤怠 | 将来拡張(requirements.md §11) | 個人・組織 |

## 2. 情報アーキテクチャ(案): ワークスペース制

現行の「スペース一覧をフラットに並べる」構成をやめ、**ワークスペース切替**
(Googleのアカウント切替/Slackのワークスペース切替のイメージ)にする。

```
アカウント(1ユーザー・1ログイン)
└ ワークスペース切替(ヘッダーのスイッチャー)
   ├ 個人ワークスペース
   │   ├ カレンダー(コア)+ 有効化したモジュール(家計簿・日記・キャリア等)
   │   └ 所属グループ(家族・友人・サークル)
   └ 組織ワークスペース(企業)… 複数所属可
       ├ 組織カレンダー・メンバー管理
       ├ グループ(部署・チーム)
       └ プロジェクト
```

- ヘッダー左上のスイッチャーで「個人 / ◯◯株式会社 / …」を切り替える。
  ナビゲーション・カレンダー・モジュールはすべて選択中ワークスペースの文脈になる
- 個人カレンダーには、共有された組織・グループの予定もレイヤーとして重ねられる(現行F-03-3を踏襲)

## 3. 変えないもの(不変条件の維持)

CLAUDE.md の不変条件はこの再構成でも**すべて維持する**:

1. デフォルト非公開 / 2. 共有は`item_shares`への参照付与のみ / 3. オープン型共有のみ
4. チャット・DM非搭載(電気通信事業法対応の恒久方針) / 5. RLS一本ルール

「企業管理」に踏み込んでも、可視性の原則(自分が作成者 or 共有先スペースのメンバー)は変えない。

## 4. 現行実装からの移行パス(実現性メモ)

現行のデータモデルは大部分がそのまま使える:

- `spaces.type`(personal/group/organization/project)と `parent_space_id` が既にあり、
  階層は「組織→プロジェクト」のみ許可する制約になっている。
  → **制約を緩めて「組織→グループ(部署)」も許可**すれば、構想の階層をそのまま表現できる
- ワークスペース切替は、**UI/ナビゲーションの再構成が主**(スイッチャー+文脈付きレイアウト)。
  スキーマ変更は小さい
- モジュール化は `spaces.settings.modules`(jsonb)に有効フラグを持たせる案。
  無効モジュールはナビに出さないだけで、データは保持
- キャリア(履歴書)は新テーブル群(学歴・職歴・資格…)+出力機能。RLSは一本ルールを適用

## 5. 名前(プロジェクト名)の再検討

コンセプトが「スケジュール軸+蓄積」に寄るため、名称変更も選択肢。
Weft(緯糸: 日々に共有を織り込む)の世界観は維持も可能。**要決定**。
