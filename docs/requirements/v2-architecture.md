# 第2弾 基盤検討: 名前・モジュール管理・インフラ(議論用・未確定)

> **ステータス: 草案**。[v2-direction.md](v2-direction.md)(方向性)の下位検討。
> 「かなり変える」前提で、先に固めるべき3点(名前 / モジュール管理 / インフラ)を比較検討する。
> 各節末尾に**推奨案**を明記。決定は Issue #11 で行う。

## 1. 名前

コンセプトが「カレンダー/スケジュールを軸に、個人と組織の記録を貯めるプラットフォーム」へ
寄るため、名称を再検討する。評価軸: コンセプト合致 / 読みやすさ(日英) / 検索性(かぶりにくさ)。

| 候補 | 由来 | 合致 | 懸念 |
|---|---|---|---|
| **Koyomi(コヨミ)** | 暦。スケジュール軸をそのまま体現 | ◎ | 一般名詞ゆえ検索性は中。ドメインは `koyomi.app` 等要確認 |
| **Tsuzuri(ツヅリ)** | 綴り。日々を綴じて蓄積する | ○ | 蓄積寄りでスケジュール感は弱め |
| **Fumi(フミ)** | 文・踏み。記録と歩みの二重 | ○ | 短すぎて検索性低 |
| **Orite(オリテ)** | 織り手。Weftの世界観を継承しつつ主体を示す | ○ | 造語ゆえ由来の説明が必要 |
| **Weft 続投** | 緯糸。個人の日々(縦糸)に共有(横糸)を織る | △〜○ | 「織物」比喩がスケジュール軸と少しずれる。一方でリポジトリ・ドキュメント資産はそのまま |

**推奨: 第一候補 Koyomi、次点 Weft続投。**
スケジュール主軸への転換を名前で宣言できるのが Koyomi。ブランド資産がまだ無い今が改名の最安タイミング。
逆に「中身の再構成に集中し、名前は市場投入前に決める」なら Weft 続投で保留するのも合理的。
決定後の作業: リポジトリ名変更(GitHub は旧URLから自動リダイレクト。作り直しは不要)、
アプリ内表記・メタデータ・ドキュメントの一括置換。

## 2. モジュール管理の方法

### 選択肢

| 方式 | 内容 | 評価 |
|---|---|---|
| **A. モノリス内モジュール(推奨)** | 1つのNext.jsアプリ内で `src/modules/<key>/` に分割し、レジストリで宣言・有効化 | 個人開発の規模に最適。ビルド・デプロイ・RLSが1系統のまま |
| B. パッケージ分割(monorepo) | pnpm workspace でモジュールを独立パッケージ化 | 将来モジュールが10超・複数人開発になったら移行。今はオーバーヘッドが勝る |
| C. マイクロフロントエンド/別アプリ | モジュールごとに別デプロイ | 認証・RLS・UXの分断コストが大きく不採用 |

### A案の具体設計

```
src/
  modules/
    calendar/   … コア(無効化不可)
    budget/     … 家計簿
    diary/      … 日記・写真
    documents/  … 文書
    career/     … キャリア(履歴書)※新規
    settlement/ … 精算
  lib/modules.ts … モジュールレジストリ(単一の正)
```

- **レジストリ**(`src/lib/modules.ts`)に全モジュールを宣言する:
  `{ key, 表示名, 対象ワークスペース種別, ナビ項目, 既定で有効か }`。
  ナビ・設定画面・ルーティングガードはすべてレジストリから生成し、直書きしない
- **有効化状態**はワークスペースの `settings.modules`(jsonb)に持つ:
  `{"budget": true, "career": false}`。無効時はナビ非表示+ルートで404(データは保持)
- **DB規約**: モジュールのテーブルは `career_` のように**モジュール名プレフィックス**を付け、
  マイグレーション・RLS(一本ルール継承)・GRANTをモジュール単位でまとめる
- **モジュール間参照**は「アイテムへのリンク(既存の双方向リンク)」経由に限定し、直接JOINしない
  (将来のパッケージ分割(B案)への退路を確保)
- 有効化の単位=将来の課金単位になり得る(requirements §11 の収益化と整合)

### 移行の考え方

現行機能を一気に `src/modules/` へ動かさない。**新規(キャリア)をこの規約で作り始め、
既存(家計簿等)はワークスペースIA再構成のタイミングで順次移設**する(ストラングラー方式)。

## 3. インフラ

### 選択肢

| 構成 | 費用感 | 強み | 弱み |
|---|---|---|---|
| **A. 現行継続: Vercel + Supabase(推奨)** | 無料枠→ Supabase Pro $25/月 + Vercel Pro $20/月 | RLS一本ルール・Auth・Storageの資産をそのまま活用。CI(ローカルSupabase)も実績あり | 無料枠は1週間非アクティブで停止(keepalive対応済み)。企業向けSSO(SAML)は上位プラン |
| B. Cloudflare(Workers + D1/Hyperdrive + R2) | 最安級 | エッジ・低コスト | **RLSモデルが使えず認可を全面自作**。移行コストが本体。不変条件5(RLS一本)と相性が悪い |
| C. VPS + セルフホストPostgres(+Supabaseセルフホスト) | 月数百〜数千円 | ロックイン最小 | 運用(バックアップ・監視・パッチ)を自分で負う。個人開発の可処分時間を食う |

**推奨: A(現行継続)。**
本プロダクトの安全性の核は「RLS一本ルール」で、これはPostgres RLS前提の設計。
Supabaseを外す選択はアーキテクチャの作り直しに等しく、第2弾の目的(IA再構成・モジュール化)と両立しない。
requirements §7 の判断(標準Postgresゆえ将来の自前移行の退路あり)も引き続き有効。

企業利用が本格化した時の拡張パス(今は着手しない):
- 認証: Supabase Auth の Google/Microsoft OAuth(無料枠で可)→ SAML SSO(Pro+アドオン)
- ストレージ費が嵩んだら写真のみ R2 へ切り出し(署名URL発行はサーバー側のまま)
- 監査要件が出たら Postgres の監査ログ拡張で対応

### リポジトリ戦略

「かなり変える」場合も**リポジトリは作り直さず現行を継続**することを推奨する。
理由: コミット履歴・CI・フック・ルール・設計書の資産が引き継がれ、GitHubのリネームは旧URLを
自動リダイレクトするため改名とも両立する。大規模変更はフェーズブランチ+PRで段階的に入れる。

## 4. 推奨のまとめ(決定待ち)

| 論点 | 推奨 |

| インフラ | **Vercel + Supabase 継続**。企業向けSSO・R2切り出しは需要が出てから |
