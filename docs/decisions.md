### アーキテクチャ上の判断

- **RLSポリシーの再帰回避**: ポリシー式がRLS付きテーブル同士を相互参照すると
  再帰評価・無限再帰エラーの恐れがあるため、メンバーシップ・所有権判定を
  `SECURITY DEFINER` のヘルパー関数(`is_space_member` / `owns_item` /
  `has_shared_access` / `shares_space_with`)に集約した。関数は boolean を
  返すのみで行データは返さない。`search_path = ''` を固定し、`anon` からの
  実行権限は剥奪済み。
- **個人スペース自動作成(F-01-3)**: `auth.users` への INSERT トリガ
  (`handle_new_user`)で profiles・spaces・space_members を同一トランザクション
  内に作成。アプリ側での作成処理は持たない(部分作成状態を作らないため)。
- **item_shares はP1でテーブル・ポリシーまで作成**: 共有UIはP3だが、
  「一本のルール」をP1時点から完全な形でテストするため(共有→可視、
  解除→不可視のDBレベル検証を実施済み)。
- **RLS検証の二段構え**:
  1. DBレベル: `scripts/local-rls-check.sh`(素のPostgreSQL+Supabaseシムで
     マイグレーション適用と分離チェック。Dockerが使えない環境向け)
  2. E2E: `e2e/rls-isolation.spec.ts`(CIで `supabase start` の上、UIとAPI
     直叩きの双方から分離を検証)

### 実装環境の制約メモ(P1)

- 開発セッションの実行環境ではDockerレジストリのblob配信CDNが
  egressポリシーで遮断されており、`supabase start` が実行不可。
  そのためE2EテストはGitHub Actions上で実行する構成とし、ローカルでは
  DBレベル検証(上記1)で代替した。

| 3 | 予定の時刻の持ち方 | occurred_on(date)+payload.start_time/end_time("HH:MM")。日またぎ予定は非対応 | カレンダーの背骨が「日付」であるため。日またぎは要望が出たら検討 |
| 4 | 収支の金額 | 正の整数(円)+kind(income/expense)で符号を表現 | 入力ミス(負数)を型で防ぐ。集計もkindで分けるだけで済む |
| 5 | 費目のカスタマイズ(F-05-2) | expense_categories テーブル+入力はdatalist(自由入力も可) | 記録済みアイテムはカテゴリ名を文字列で保持するため、費目を消しても記録は壊れない |
| 6 | タスク(個人) | type=task、payload.status(todo/doing/done)のみ。期限=occurred_on | §4.3のpersonalタスクの最小形。担当者等はP5のプロジェクトタスクで |
| 7 | 双方向リンクの格納 | 無向。item_id_a < item_id_b に正規化して1行 | 重複防止が主キーだけで済み、「AからもBからも見える」をクエリ2条件で実現 |
| 8 | リンクの可視性(F-09-5) | 両端とも閲覧できる場合のみリンク行が見える | 片端でも見えないと存在自体を隠す、を最も安全側に倒した |
| 9 | 月次集計 | SECURITY INVOKER のRPC(expense_monthly_summary) | §6.2「集計はビューまたはRPC」。INVOKERなのでRLSがそのまま効く |
| 10 | デザインモック承認の省略 | ユーザー指示「全て開発を完了」により、P2冒頭のモック承認は省略しCLAUDE.mdの原則を直接適用 | 配色(生成り/墨/藍1色)・明朝見出し・罫線・文具的マイクロコピー(「記す」「破り捨てる」「ほどく」等)で実装 |

### メモ

- 週表示はリスト型(モバイルファースト優先)。時間軸グリッドは要望があれば検討
- カレンダーのドット凡例: 予定=藍 / 日記=墨 / 収支=薄墨 / つとめ=白抜き

## P3 共有(2026-07-19)

### 仮決め事項

| # | 迷った点 | 仮決め内容 | 理由 |
|---|---|---|---|
| 1 | 招待リンクの有効期限・回数(§13未決) | 期限7日・回数無制限(期限内なら誰でも参加可)。手動で「破る」ことも可能 | 小規模グループの想定では期限だけで十分。回数制限は要望が出たら追加 |
| 2 | 招待トークンの生成 | gen_random_uuid のハイフン除去(122bitランダム) | pgcrypto拡張に依存せずPostgreSQLコアのみで十分な強度 |
| 3 | グループ作成の実装 | SECURITY DEFINER の RPC(create_group) | spaces へのINSERTとowner登録を1トランザクションで行い、部分状態を作らない |
| 4 | 招待受諾 | SECURITY DEFINER の RPC(accept_invitation)。トークン自体は非メンバーには読めない | 未メンバーは space_members に行を足せないため。トークン検証はDB内で完結 |
| 5 | owner の退出 | 不可(UI・RLSとも)。除名対象にも owner はなれない | グループが世話役不在で宙に浮くのを防ぐ。owner移譲は将来検討(ISSUE相当) |
| 6 | コメントの編集 | 編集なし・削除のみ(本人) | 回覧板のメタファー(書いたものは残る)。最小実装 |
| 7 | リアクションのスタンプセット | 固定5種(🌸/👏/祝/見た/🍵) | 絵文字の多用を避けるデザイン原則との折衷。文字スタンプ(祝・見た)で和の趣 |
| 8 | カレンダーのレイヤー帰属 | 自分の記録=「わたし」レイヤー、他人の記録=共有元スペース(複数共有時は最初の1つ) | 同じアイテムを二重に描かないための簡略化。厳密な多重帰属は要望が出たら |
| 9 | コメント・リアクションの前提 | 「そのスペースへ共有中のアイテム」にのみ書ける(RLSで強制) | 共有解除後に新規コメントが付く事故を防ぐ。解除済みアイテムの既存コメントは残るが、アイテム本体が見えないため実質到達不能 |
| 10 | フィードの内容 | item_shares の時系列(=差し出しの記録) | F-07-3「共有・更新を時系列で」の最小形。更新イベントの追跡はP6以降で検討 |

## P5 組織・プロジェクト(2026-07-19)

### 仮決め事項

| # | 迷った点 | 仮決め内容 | 理由 |
|---|---|---|---|
| 1 | プロジェクトのアイテムの持ち方 | 個人所有アイテム(origin=プロジェクト)+作成時に自動で item_shares へ共有 | 「共有=参照付与」の一本のルールを崩さない(不変条件2・5)。タスク・ナレッジも例外にしない |
| 2 | 組織メンバーへのプロジェクト可視性 | 組織の owner/admin はプロジェクトの「名前とメタ情報(予実・状態)」のみ見える。中のアイテムは見えない | F-08-5(ダッシュボード)と§6.3(プロジェクト単位の分離)の両立 |
| 3 | ダッシュボードの実績値 | 参加しているプロジェクトのみ表示。非参加は「参加者のみ閲覧できます」 | 実績はアイテム(RLS対象)の集計のため。集計だけ definer で見せるのは §6.3 の精神に反すると判断 |
| 4 | タスクのステータス変更 | SECURITY DEFINER RPC(update_task_status)で「作成者+担当者」のみ許可。本文の update は作成者のみのまま | §6.3の唯一の例外を、payloadのstatusキー書き換えに限定して実装 |
| 5 | プロジェクトの実績(予実の「実」) | そのプロジェクトへ共有された expense アイテムの合算(SECURITY INVOKER RPC) | F-05-4「収支をプロジェクトへリンク」を共有として表現。個人の家計簿と二重管理しない |
| 6 | プロジェクトメンバーの追加 | 親組織のメンバーを owner/admin が直接追加(RPC)。招待リンクも併用可 | 業務利用では招待リンクの往復が煩雑なため |
| 7 | カンバン表示 | ステータス別の縦積みリスト(モバイルファースト) | F-08-2「カンバンまたはリスト」。375px幅では横カンバンが成立しない |
| 8 | 費目別予算の period | カラムは用意、P5のUIでは全期間(null)のみ | 月次予算の要望が出たらUI追加。スキーマ変更を避けるため先に用意 |

### 公開前セキュリティ自己点検(§8.4 / §9)

- [x] 全テーブルRLS有効・anonはテーブルGRANTなし(未ログインは何も読めない)
- [x] service_role キーはサーバー専用モジュールのみ(server-only)。クライアントコードに露出なし
- [x] アクセス制御はRLS一本。クライアント側の出し分けに依存した箇所なし
- [x] 共有状態は常にバッジ・差し出し先一覧で可視(F-06-4)
- [x] チャット・DM・非公開返信なし(§9)。コメント・リアクションは全員可視のオープン型のみ
- [x] 精算は記録と計算のみ(送金なし)を規約・画面の両方に明記
- [x] 退会で本人起点の全データ+Storageの写真を削除
- [x] ログインの next リダイレクトは自サイトパスのみ許可(オープンリダイレクト対策)
- [x] 招待トークンは122bitランダム・期限付き・非メンバーには読めない
- [x] 一覧APIはすべてページネーション付き(品質基準)

### 公開時に運営者が行うこと(コード外の作業)

1. 本番Supabaseプロジェクト作成 → `supabase db push` でマイグレーション適用
2. Vercel に環境変数(NEXT_PUBLIC_SUPABASE_URL / NEXT_PUBLIC_SUPABASE_ANON_KEY / SUPABASE_SERVICE_ROLE_KEY)を設定
3. GitHub Secrets に SUPABASE_URL / SUPABASE_ANON_KEY を設定(keepalive用)
4. Supabase Auth 設定: メール確認をオン、サイトURLを本番ドメインに
5. (任意)Google OAuth の設定 → F-01-2 の実装を依頼
6. 一般公開後は CLAUDE.md の運用を「PRレビュー(人間の承認必須)」へ切替
